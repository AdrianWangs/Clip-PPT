# 研究意义：隐私保护与智能检索的完美结合

- **云计算的普及**：企业和个人越来越多地将多媒体数据外包到云平台，享受强大的存储和计算能力。
- **隐私泄露风险**：传统明文跨模态检索在云环境中不可避免地导致隐私泄露。
- **技术空白**：现有可搜索加密方案主要解决文本到文本或图像到图像检索，跨模态检索的隐私保护研究相对空白。
- **CLIP的启发**：CLIP模型的成功为跨模态理解提供了强大的特征提取能力。
- **创新结合**：本论文首次将CLIP、局部敏感哈希(LSH)和对称可搜索加密(SSE)相结合，实现了安全高效的跨模态检索。

## 云端跨模态检索面临的挑战

::left::

### 隐私保护的迫切需求

- **数据外包风险**：用户将图像和文本数据外包到云端，面临隐私泄露风险
- **检索过程暴露**：传统检索会暴露用户查询意图和数据内容
- **法律法规要求**：GDPR等法规对数据隐私保护提出严格要求
- **用户信任危机**：隐私泄露事件频发，用户对云服务的信任度下降

*云端数据处理的便利性与隐私保护之间存在根本性矛盾。*

::right::

### 技术挑战与局限

- **现有方案局限**：大多数可搜索加密方案仅支持单模态检索
- **效率与准确性平衡**：加密处理往往导致检索效率和准确性下降
- **密钥管理复杂**：多用户环境下的密钥管理负担沉重
- **语义对齐困难**：加密状态下难以实现跨模态语义对齐

*需要一种既能保护隐私又能保持高效准确检索的创新方案。*

## 现有方案的不足

<br>

### 主要问题分析

- **SCMR方案**：使用Paillier同态加密，但服务器端匹配过程引入的随机性导致查询结果准确性较低
- **PPCMR方案**：采用转置密码加密，但密钥数量等于用户文件数量，密钥管理负担沉重；难以在加密状态下对齐图文语义
- **PITR方案**：使用HNSW算法生成多层图结构索引，用户本地计算负担大，不适用于大数据场景

<br>

**本文目标：** <u>设计一个高效、安全、实用的云端跨模态检索方案，在保护隐私的同时实现准确快速的检索。</u>

<!--

## 核心技术组件

给定用户图像集 $X = \{x_i\}_{i=1}^n$ 和查询文本 $q$：

::left::

### CLIP特征提取
- **图像特征**：$f_i = \text{CLIP.Extract}(x_i)$
- **文本特征**：$f_q = \text{CLIP.Extract}(q)$
- **语义对齐**：映射到统一512维向量空间
- **关键作用**：消除跨模态数据的语义鸿沟

### LSH哈希分桶
- **哈希函数**：$H(f) = \lfloor\frac{A \times f + b}{r}\rfloor$
- **参数设置**：$A \in \mathbb{R}^{128 \times 512}$，$B=128$
- **分桶策略**：将128位哈希码分为64段，每段2位
- **快速过滤**：将相似特征映射到同哈希桶，大幅减少候选集

::right::

### 两层加密索引架构
- **第一层索引**：SSE加密的倒排索引（用于快速筛选）
- **第二层索引**：Secure kNN加密的原始特征（用于精确排序）
- **并行处理**：64个SSE索引可同时并行搜索
- **隐私保护**：加密状态下完成所有检索计算

### 工作流程概览
**完整流程：** 本地CLIP特征提取 → LSH哈希分桶 → SSE加密倒排索引 → Secure kNN加密特征 → 云端上传 → 加密检索 → 结果解密

## 局部敏感哈希 (LSH)

::left::

### E2LSH算法原理

**哈希函数：**
$$H(f) = \left\lfloor\frac{A \times f + b}{r}\right\rfloor$$

**参数详细说明：**
- $A \in \mathbb{R}^{B \times D}$：满足标准正态分布的随机投影矩阵
- $b$：$[0,r]$区间内均匀分布的随机偏移向量
- $r$：分桶宽度参数，控制哈希分桶粒度
- $B=128$：生成的哈希码总长度
- $D=512$：输入特征向量维度

::right::

### 哈希码分段与倒排索引构建

**分段策略：**
- 将128位哈希码平均分为64个独立段
- 每段长度：$\lfloor\frac{128}{64}\rfloor = 2$位
- 为每段独立构建一个倒排索引：$I = \{I_j\}_{j=1}^{64}$
- 索引关键字：第j段的哈希值
- 索引值：具有相同哈希值的所有图像ID集合

## 对称可搜索加密 (SSE) 与安全kNN

::left::

### SSE第一层索引：快速筛选

**核心功能：**
- **密钥生成：** $k_{SSE} = \{k_{SSE,j}\}_{j=1}^{64}$（为每个索引生成独立密钥）
- **索引加密：** $\tilde{I}_j^1 = \text{SSE.BuildIndex}(I_j, k_{SSE,j})$
- **陷阱门生成：** $\tilde{T}_j^1 = \text{SSE.GenTrapdoor}(h_{q,j}, k_{SSE,j})$
- **并行搜索：** 云服务器可并行搜索64个SSE索引

**安全特性：**
只泄露搜索模式（相同查询可被识别），但不会泄露查询的具体语义内容

::right::

### Secure kNN第二层索引：精确排序

**密钥组成：**
$$k_{kNN} = \{M_1, M_2, s\}$$  
（两个随机可逆矩阵和一个随机二进制向量）

**加密过程：**
- **特征向量分裂：** 根据向量s将原特征分为两个随机向量
- **特征加密：** $\tilde{f}_i = \{M_1^T f_{i,1}, M_2^T f_{i,2}\}$
- **查询加密：** $\tilde{q} = \{M_1^{-1} q_1, M_2^{-1} q_2\}$

**内积保持特性：**
$$\text{Sim}(\tilde{f}_i, \tilde{q}) = \tilde{f}_i \cdot \tilde{q} = f_i \cdot q$$

<div class="mt-4 p-3 bg-gray-100 rounded text-sm">
<strong>两层协同工作机制：</strong>第一层SSE快速筛选出潜在候选集（通常包含约10%-20%的总数据），第二层Secure kNN在候选集内进行精确相似度计算和排序。这种设计完美平衡了检索效率与准确性。
</div>

<!--

第二层采用安全k近邻(Secure kNN)算法：该算法通过随机矩阵分裂技术保护向量隐私，同时巧妙地保持了内积不变性。具体来说，每个特征向量会被随机分裂成两个部分，再分别用不同矩阵加密。查询向量也经过类似处理。最精妙之处在于，加密向量的内积等于原始向量的内积，这使得可以在加密状态下直接计算相似度。

这种两层设计实现了效率与准确性的完美平衡：SSE快速缩小搜索范围，Secure kNN保证排序精度，同时整个过程中云服务器始终无法获取明文信息。
-->

## Secure kNN算法流程详解

::left::

### 密钥构成
$$k_{kNN} = \{M_1, M_2, s\}$$
其中：
- $M_1, M_2 \in \mathbb{R}^{d \times d}$ 为随机可逆矩阵
- $s \in \{0,1\}^d$ 为随机二进制向量

1. **向量分裂**：根据s将原特征向量$f_i$分为两个随机向量$f_{i,1}, f_{i,2}$
   - 当$s[j] = 0$时：$f_{i,1}[j] = f_{i,2}[j] = f_i[j]$
   - 当$s[j] = 1$时：随机选择$f_{i,1}[j]$和$f_{i,2}[j]$使$f_{i,1}[j] + f_{i,2}[j] = f_i[j]$
2. **矩阵变换**：$\tilde{f}_i = \{M_1^T f_{i,1}, M_2^T f_{i,2}\}$

::right::

### 查询向量加密步骤
1. **向量分裂**：根据s将查询向量$q$分为两个随机向量$q_1, q_2$
   - 当$s[j] = 1$时：$q_1[j] = q_2[j] = q[j]$
   - 当$s[j] = 0$时：随机选择$q_1[j]$和$q_2[j]$使$q_1[j] + q_2[j] = q[j]$
2. **矩阵变换**：$\tilde{q} = \{M_1^{-1} q_1, M_2^{-1} q_2\}$

<div class="mt-4 p-3 bg-gray-100 rounded text-sm">
<strong>关键设计：</strong>特征向量和查询向量的分裂规则基于相同的二进制向量s，但分裂模式相反，这是内积保持性的关键！
</div>

<!--
现在让我们详细解析Secure kNN的加密流程。这是该方案中保障隐私的核心机制。

首先，Secure kNN的密钥由三个部分组成：两个随机的d×d可逆矩阵M₁和M₂，以及一个随机的d维二进制向量s。

在加密特征向量时，我们分两步进行。第一步是向量分裂，根据随机二进制向量s将原特征向量fᵢ分裂成两个随机向量fᵢ₁和fᵢ₂。分裂规则很巧妙：对于向量中的每个元素位置j，如果s[j]=0，那么两个分裂后的向量在该位置的值都等于原向量的值；如果s[j]=1，那么在该位置随机选择两个数，只要它们的和等于原向量的值即可。第二步是矩阵变换，将这两个分裂后的向量分别用矩阵M₁和M₂的转置进行变换，得到最终的加密特征向量。

对于查询向量的加密，分裂规则与特征向量相反：如果s[j]=1，则两个分裂后的向量在该位置的值都等于原向量的值；如果s[j]=0，则随机选择两个数，只要它们的和等于原向量的值。然后，将这两个分裂后的向量分别用矩阵M₁和M₂的逆进行变换，得到最终的加密查询向量。

这种设计的妙处在于，特征向量和查询向量的分裂规则虽然基于同一个二进制向量s，但它们的分裂模式是相反的。正是这种相反的分裂模式，为接下来要证明的内积保持性奠定了基础。这种设计既保证了数据的安全性，又为后续的相似度计算提供了可能。
-->

## 内积保持性原理

$$
\begin{align}
\text{Sim}(\tilde{f}_i, \tilde{q}) &= \tilde{f}_i \cdot \tilde{q} \\
&= M_1^T f_{i,1} \cdot M_1^{-1} q_1 + M_2^T f_{i,2} \cdot M_2^{-1} q_2 \\
&= f_{i,1}^T M_1 M_1^{-1} q_1 + f_{i,2}^T M_2 M_2^{-1} q_2 \\
&= f_{i,1}^T q_1 + f_{i,2}^T q_2
\end{align}
$$

::left::



**情况一：当$s[j] = 0$时**
- 特征向量分裂：$f_{i,1}[j] = f_{i,2}[j] = f_i[j]$
- 查询向量分裂：$q_1[j] + q_2[j] = q[j]$
- 内积贡献：$f_{i,1}[j]q_1[j] + f_{i,2}[j]q_2[j] = f_i[j](q_1[j] + q_2[j]) = f_i[j]q[j]$

<br>

> $$综上所述：\text{Sim}(\tilde{f}_i, \tilde{q}) = f_{i,1}^T q_1 + f_{i,2}^T q_2 = f_i^T q = f_i \cdot q$$

::right::

**情况二：当$s[j] = 1$时**
- 特征向量分裂：$f_{i,1}[j] + f_{i,2}[j] = f_i[j]$
- 查询向量分裂：$q_1[j] = q_2[j] = q[j]$
- 内积贡献：$f_{i,1}[j]q_1[j] + f_{i,2}[j]q_2[j] = q[j](f_{i,1}[j] + f_{i,2}[j]) = f_i[j]q[j]$

<!--
现在，让我们通过数学推导来证明Secure kNN算法的内积保持特性。这是整个方案中最精妙的数学设计。

首先，我们计算加密后的特征向量和查询向量的内积。这个内积可以分解为两个部分的和：第一部分是M₁^T fᵢ₁与M₁⁻¹ q₁的内积，第二部分是M₂^T fᵢ₂与M₂⁻¹ q₂的内积。

由于M₁和M₂都是可逆矩阵，所以M₁ M₁⁻¹ = I（单位矩阵），同样M₂ M₂⁻¹ = I。因此，加密向量的内积进一步简化为fᵢ₁^T q₁ + fᵢ₂^T q₂。

接下来，我们分两种情况验证这个表达式等于原向量的内积：

当s[j]=0时，特征向量在该位置的两个分裂值相等，都等于原特征值；而查询向量在该位置的两个分裂值之和等于原查询值。因此，该位置对总内积的贡献等于原特征值乘以原查询值。

当s[j]=1时，查询向量在该位置的两个分裂值相等，都等于原查询值；而特征向量在该位置的两个分裂值之和等于原特征值。因此，该位置对总内积的贡献也等于原特征值乘以原查询值。

## 完整的检索流程

::left::

### 索引构建阶段（数据所有者执行）
1. **特征提取**：CLIP提取图像特征向量
2. **LSH处理**：生成128位哈希码并分64段
3. **索引构建**：为每段哈希构建倒排索引
4. **两层加密**：SSE加密倒排索引 + Secure kNN加密原始特征
5. **数据上传**：上传加密图像及索引至云端

<br>

### 查询检索阶段（用户执行）
1. **特征提取**：CLIP提取查询文本特征
2. **陷阱门生成**：生成SSE陷阱门和Secure kNN加密查询
3. **云端检索**：并行搜索SSE索引获候选集，再用Secure kNN精确排序
4. **结果返回**：获取并解密Top-k个最相似图像

<h4 class="font-semibold text-gray-800 mb-1">参数调优复杂</h4>
<p class="text-sm mb-3">LSH参数(B, r, m)的选择需要在效率和精确率间权衡，调优复杂</p>

<h4 class="font-semibold text-gray-800 mb-1">CLIP模型依赖</h4>
