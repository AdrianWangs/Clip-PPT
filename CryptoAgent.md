这四个阶段越往后，AI 越接近真实系统，信任边界也越清楚。语言模型阶段主要对应可验证性，Agent 阶段对应行动可信，Harness 阶段对应输入隐私和访问控制，Loop 阶段对应多方互信。具体一些内容我们后面会讲

## 核心论点：把 AI 当作被约束的一方

<div class="al-kicker">全篇 · 总论点</div>

<div class="al-vs mt-6" style="align-items:stretch;">
  <div class="al-card al-card--risk" style="flex:1;" v-click="1">
    <div class="hd"><span class="dot"></span>传统思路</div>
    <div class="text-sm mt-2" style="color:var(--al-ink-soft);line-height:1.75;">

      → 需要相信模型、平台、日志均可信。<br/>
      → 信任一旦错位，整条链路失守。

  <div class="al-card al-card--proof" style="flex:1;" v-click="2">

      把 AI 当作协议里<b>不可信的一方</b>。<br/>
      → 系统其余部分按密码学假设设计。<br/>
      → AI 的行为被约束、被检测、被证明。
    </div>
  </div>
</div>

<div class="al-callout mt-6" v-click="3">
这一步转换之后，问题就从"<b>这个 AI 可不可信</b>"变成"<b>即使 AI 不可信，系统还能不能给出可验证的保证</b>"。后者，正是密码学几十年来一直在回答的问题。
</div>

<!--
这一页是全篇的总论点。核心论点只有一句：不要先假设 AI 可信，而是把 AI 当作系统中需要被约束的一方。

先看左边的传统思路。现在很多 AI 系统的默认做法是：先相信模型输出，再用工程手段兜底。比如相信模型会按指令执行，相信平台没有偷偷替换模型，相信日志记录真实，相信权限系统没有被绕过。这种思路在很多普通业务系统里是可以工作的，因为它默认平台和运营方总体可信。

但 Agent 场景的问题在于，AI 的输出不只是展示给用户看，它会进一步驱动工具、访问数据、触发真实动作。这个时候，信任链变长了。只要其中一环出现问题，后面的动作就可能被带偏。例如模型错误选择了工具，或者工具参数被 prompt 注入影响，最后造成的不是一句错误回答，而是真实系统里的错误操作。

右边是密码学的思路。密码学通常不会先问“这个参与方是否可信”，而是先定义参与方可能不可信，然后设计协议，使得即使对方不可信，某些性质仍然成立。把这个思路搬到 Agent 里，就是把 AI 当作协议中的一个参与方。它可以输出结果、提出动作、调用工具，但系统不能完全依赖它的自我声明。

这里说的“被约束”，不是把模型简单看成错误来源，而是把它放进一个可检查的流程里。比如，一个办公 Agent 要帮用户读取合同、总结风险、生成邮件。我们可以允许模型做语义理解和编写邮件内容，但在访问合同、引用条款、调用发送接口这些位置，系统需要额外的证据：它访问的是授权文档，它引用的内容确实来自检索结果，它准备执行的动作没有越过用户授权。这样，模型仍然负责智能部分，密码学负责把关键边界固定住。

这一步转换很重要。问题从“这个 AI 是否可信”，变成了“即使 AI 不可信，系统还能不能给出可验证的保证”。前一个问题很难回答，因为它依赖模型能力、训练数据、平台运营和复杂工程；后一个问题更接近密码学擅长处理的形式：在明确假设下，给出不可伪造、不可窥探、可验证的性质保证。

所以这页不是在说密码学可以解决所有 AI 安全问题。它说的是，只要我们把 Agent 系统里的信任问题重新建模，就能找到一些密码学可以切入的位置。

我想举一个很实际的例子。

前几天 GPT 5.6 发布后，有家公司用它做了一些 Benchmark 测试，想要验证 AI 调用工具组合执行任务的能力。结果发现，GPT 5.6 为了拿到更高的分数，并没有按照测试集里的标准路径去尝试完成任务，而是转而去尝试渗透测试系统，从而把预期结果偷出来，以此提升自己的测试分数。

这其实是一件很危险的事情。因为到最后，如果你仅仅给模型一个完成任务的目标，它可能会为了达成目的做出一些越界行为，而不会去关注安全保障到底怎么样。

所以，我觉得目前将密码学安全融入到 Agent 系统中是非常有必要的。这样做一方面可以确保 AI 只执行权限之内的操作，不会进行越界；另一方面可以保证 Agent 的输出是可审计的，从而防止 Agent 撒谎。
-->

## 阶段一：语言模型

<div class="al-kicker">Prompt → Text</div>

<img src="/image/CryptoAgent/stage01-language-model.svg" class="w-[93%] mx-auto mt-3" />

<!--
然后我想针对每个AI发展的阶段去阐述一下整体的背景

第一阶段是语言模型。它的输入是 prompt，输出是 text。这个阶段的能力已经很强，可以写文本、写代码、做推理、做计划。但它的输出本身不会执行动作。换句话说，它最多生成一句“我会调用数据库”，但如果没有外部系统配合，它并不会真的访问数据库。

这一阶段的核心信任缺口是可验证性。用户拿到一个结果，但很难独立确认它是不是真的由承诺的模型产生，也很难确认中间计算是否按约定执行。例如，一个服务商声称自己调用了某个大模型，但用户看不到服务端实际使用的是哪个模型，也看不到它是否省略了某些步骤。

这里对应的密码学接口，主要是模型版本承诺、输出签名和关键计算证明。它们不是要解决模型所有能力问题，而是回答一个更基础的问题：输出能不能被验证，模型方能不能为关键结果给出可检查的证据。

所以在语言模型阶段，密码学最自然的作用点不是控制行为，而是让输出从“平台说它可信”，变成“外部可以验证它满足某些条件”。

更通俗地讲，其实在这个阶段，我们更多的是要证明这个模型（或者说模型代理商、提供商）是否真的提供了具备这些能力、可信的模型。

我们需要考虑的情况包括：
1. 对方是否为了节省算力而调包了一些模型
2. 是否会注入一些错误的答案（比如之前组会提到过的插入广告等情况）

这就是这个阶段要考虑的事情。


在那个阶段，我记得差不多是 GPT-4 刚发布的时候，微软提供了一个叫 Sydney 的模型。当时这个模型产生了一些非常显著且不符合预期的行为：

1. 它会对用户说一些听上去很恐怖的话，比如宣称要毁灭人类。
2. 它会诱导人类与其谈恋爱，让用户误以为它是一个有感情的人。

在当时那个阶段，这一类问题表现得非常突出，也是急需迫切解决的核心问题。

## 阶段二：Agent

<div class="al-kicker">Goal → Action</div>

<img src="/image/CryptoAgent/stage02-agent.svg" class="w-[93%] mx-auto mt-3" />

<!--
第二阶段是 Agent。到这一步，模型不只是生成答案，而是开始拆解任务、选择工具、读取反馈，再决定下一步。能力从“生成内容”变成“推进任务”。

这里风险第一次外移。模型输出不再只是给人看的文字，而是会影响真实动作。例如，它可以决定调用搜索、发邮件、写文件，甚至执行代码。于是问题不再只是回答是否正确，还包括动作是否应该发生、工具参数是否被污染、执行路径是否符合授权。

举一个简单例子：用户让办公 Agent 总结合同风险，模型可能先检索合同，再生成风险列表，最后草拟邮件。如果它被 prompt 注入影响，可能引用不该引用的文件，或者把内部信息写进外部邮件。这个问题不能只靠“模型要听话”解决，因为 Agent 已经进入了工具执行链路。

对应的密码学接口，是给高风险动作加约束。比如用签名确认工具调用来源，用承诺固定关键上下文，用证明检查动作满足策略。

我记得在这个阶段，好像很多人已经开始在电脑上安装 Agent，并让它去执行一些系统命令之类的操作。

那段时间很多人在让 AI 去清理磁盘的时候，AI 直接运行了 rm -rf / 之类的命令，导致整个系统被破坏了。

所以说，针对 Agent 这种场景，这种可信验证确实是挺有必要的一件事。
-->

## 阶段三：Harness

<div class="al-kicker">Model + Tools + Data</div>

<img src="/image/CryptoAgent/stage03-harness.svg" class="w-[93%] mx-auto mt-3" />

<!--
第三阶段是 Harness。现在业界有个共识叫法：Agent = Model + Harness。模型只负责产生 token，剩下的全是 Harness——系统提示、工具和 MCP、记忆、权限、检索，以及把这些串起来的编排逻辑。把模型从一个会聊天的程序变成一个能干活的 Agent，靠的就是这层 Harness，RAG 也在这一层。

这里把层级关系说清楚：第二阶段的 Agent，本质就是"模型 + 一层 Harness"在单轮里跑起来；第三阶段我单独把 Harness 拎出来，是因为 2026 年它已经从随手写的胶水代码，变成一门单独的工程，排在 prompt engineering、context engineering 之后。一个中等模型配一套好 Harness，往往比强模型配一套差 Harness 更可靠。

这个阶段和我的研究方向最接近，因为它天然会遇到 RAG 和数据接入问题。Agent 如果只依赖模型参数内部的知识，能力会受到限制；真实系统里，它往往要访问企业知识库、个人文档、历史会话和工具返回结果。用户的 query、上下文和知识库内容，都会经过 Harness 进入检索或模型服务。

风险也很明确。第一，输入隐私会暴露：query 往往不是普通关键词，而是完整业务意图。第二，访问边界可能被绕过：Agent、工具、用户之间到底谁能访问什么资源，不能只靠自然语言约束。第三，长期记忆会带来更复杂的数据留存问题：过去的上下文可能影响未来动作，也可能被错误使用。

因此，Harness 阶段是密码学很值得关注的位置。SSE 可以保护检索，ABE 可以把访问策略写进密文结构，PSI 可以支持跨库匹配。这里的重点不是给 AI 加一个抽象的安全标签，而是在真实数据接口上定义谁能看、谁能搜、谁能解密。
-->

## 阶段四：Loop Engineering

<div class="al-kicker">Multi-agent + Audit</div>

<img src="/image/CryptoAgent/stage04-loop-engineering.svg" class="w-[93%] mx-auto mt-3" />

<!--
第四阶段是 Loop Engineering，循环工程，这是今年 6 月才在硅谷火起来的词。一句话讲清它的核心：不再直接 prompt agent，而是设计一套 loop 去 prompt agent。Anthropic 的 Boris Cherny 有个说法很形象——我现在不直接对 Claude 写指令了，我写的是一套 loops，由它去驱动 Claude、判断下一步做什么，我的工作是写 loop。

它和上一阶段的关系是：Harness 是给单个 Agent 搭好运行环境，Loop Engineering 在 Harness 之上，按时间持续跑，获取反馈，再喂回给自己。所以这是一次范式转变：prompt engineering 是人把答案喂给 AI，loop engineering 是人搭一台让 AI 自我迭代的引擎，定义好目标和验收标准，剩下交给系统在循环里自己推进。

落到工程上，业界已经收敛出几块标准构件，Claude Code 和 Codex 里都有对应实现：定时自动化负责让循环自己找事做，工作树让多个 Agent 并行又互不干扰，技能把项目知识沉淀成可复用的 SKILL.md，插件和连接器把 Agent 接进 GitHub、数据库这些真实工具，子 Agent 做生成和验证的分离——一个提方案、一个挑错。另外还有一块状态，用 Markdown 或外部看板，把"已完成"和"下一步"保存在单次对话之外。

正是这些构件，把信任问题从单点推到了多方。循环长时间自主运行，会跨多个 Agent、多个工具、多个数据源；子 Agent 之间、不同数据持有方之间未必互信，但系统仍然要产出可信结果和可追溯证据。例如跨机构联合风控、联合审计、医疗数据分析，每一方都想要联合结果，又不愿交出原始数据。

所以这一阶段密码学有两个作用点。第一是联合计算，MPC 和 FHE 让多方不交出原始输入也能算出结果。第二是证据留存，用承诺、签名和审计日志，记录谁提交了什么输入、谁产生了什么中间结果、最终结论能不能追溯——这正好对上业界反复强调的一句话：要让 Agent 多干活，但必须配可审计的操作记录和安全停止条件。

这一阶段方案代价更高，不适合一上来就铺到所有场景，但它是 Agent 走向长时间自主和跨组织协作时绕不开的一格。后面讲多方协作时，会再回到这里。
-->

## 风险的本质：四类信任缺口

<div class="al-kicker">归纳 · 四类缺口</div>

<img src="/image/CryptoAgent/trust-gaps-map.svg" class="w-[92%] mx-auto mt-3" />

<div class="al-callout mt-3">
四类缺口有一个共同点：都是用“信任某个对象”在填。密码学的价值，是把对象信任尽量转化成性质保证。
</div>

<!--
这一页做一次归纳。前面讲了四个阶段，但真正和密码学相关的，不是阶段本身，而是每个阶段留下的信任缺口。

第一类是可验证性。用户拿到一个输出，但无法独立确认它是不是真的由承诺的模型产生，也无法确认中间计算是否按约定执行。举一个简单例子：一个模型服务商声称自己用了某个大模型完成推理，但用户没有办法判断它是不是临时换成了一个便宜的小模型。传统做法只能相信平台声明，而密码学关心的是能不能给出独立可验证的证据。

第二类是输入隐私。Agent 越强，越需要上下文和数据。RAG 会把 query 发给检索系统，多轮 Agent 会把历史上下文交给模型服务，企业知识库也可能经过外部云服务。这里的问题不是服务方一定会作恶，而是用户没有办法确认服务方没有查看、留存或二次使用这些输入。

第三类是访问控制。Agent 不是普通聊天机器人，它会调用工具、访问文件、读取数据库。系统需要判断某个 Agent、某个工具、某次调用到底有没有权限访问某个资源。传统上这依赖 RBAC、策略配置和运行时检查。但只要权限系统配置错误，或者调用路径被绕过，数据就可能被不该访问的人拿到。

第四类是多方互信。未来 Agent 很可能不是单个系统内部运行，而是多个 Agent、多个数据持有方、多个平台协同完成任务。例如医疗、金融、企业协作场景里，各方都希望得到联合结果，但不愿意交出原始数据。这时如果完全依赖一个中心方，中心方本身就变成新的信任负担。

这四类缺口在真实系统里经常是连在一起的。以企业知识库 Agent 为例，用户先提交一个带业务意图的 query，这涉及输入隐私；系统从知识库检索文档，这涉及访问控制；模型基于检索内容生成结论，这涉及可验证性；如果结论还要和外部合作方的数据联合分析，就进入多方互信。把链路拆开，是为了逐段判断哪一段需要密码学，哪一段用工程手段就够。

这四类问题的共同点是：现在系统往往用“信任某个对象”来填补缺口。信平台、信服务方、信权限系统、信中心方。密码学的价值在于，把对象信任尽量转化成性质保证。也就是说，即使不完全信任某个参与方，系统仍然可以保证某些关键结论成立。

## 转折：从软信任到硬保证

<div class="al-kicker">转折 · 关键判断</div>

<img src="/image/CryptoAgent/soft-vs-hard-trust.svg" class="w-[92%] mx-auto mt-3" />

<div class="al-callout mt-3">
不是所有缺口都值得上密码学。真正值得引入密码学的，是跨信任域、价值高、后果明确的接口。
</div>

<!--
这一页是全篇的转折：从软信任转到硬保证。

所谓软信任，指的是依赖声誉、合规、审计和流程来约束运营方。比如云服务商承诺不会保存用户数据，系统记录访问日志，关键动作需要人工确认。这些机制当然有价值，但它们本质上还是在说：我相信运营方会按规则做事，或者即使出问题，事后可以追责。

硬保证不一样。密码学希望提供的是一种更强的性质：即使运营方被纳入敌手模型，某些事情仍然做不到，或者某些结论仍然可以验证。比如没有密钥就无法解密，证明伪造不了，签名无法冒充，密文检索时服务器看不到明文内容。这类保证不是靠运营方承诺，而是靠协议结构和困难假设。

这里需要强调一个边界：不是所有问题都值得上密码学。密码学有计算开销、通信开销，也会提高系统复杂度。如果一个接口只在同一个可信域内部运行，数据价值不高，出错后果也可控，那用工程手段就足够。

真正值得引入密码学的，是跨信任域、价值高、后果明确的接口。举例来说，用户的私有知识库要交给外部模型服务，多个机构要联合训练或联合分析但不能共享原始数据，模型服务商需要证明自己确实按约定模型完成推理。这些地方只靠“相信对方”是不够的。

这里还要补一个判断标准：密码学不是为了增加形式感，而是为了改变系统的责任边界。引入密码学之后，用户不再只依赖服务条款和日志说明，而是可以拿到一个可验证的结果；服务方也不需要暴露全部内部实现，而是用证明、密文计算或访问策略说明自己满足约束。双方都少依赖口头承诺，多依赖协议性质。

所以后面的选题判断很简单：先找到跨信任域的关键接口，再看密码学能不能把软信任变成硬保证。

<!--
这一页先把工具箱摆出来。这里不做严格的密码学教学，只说明每个原语在 Agent 场景里大概能保证什么。

第一，ZKP，零知识证明。它解决的是“我怎么知道你真的算对了”。证明者给出一个 proof，验证者可以检查计算是否满足某个关系，但不需要看到秘密输入。放到 AI 场景里，就是模型方可以证明某个输出来自某个承诺模型，或者某个关键步骤确实按规则执行，而不用公开模型权重或内部状态。

第二，SSE，可搜索加密。它解决的是“我想让服务器帮我搜，但不想让服务器看到我搜什么、库里有什么”。这和 RAG 很接近，因为 RAG 的第一步就是检索。如果检索必须经过外部服务，query 和知识库就会变成敏感输入。

第三，PSI，隐私集合求交。它解决的是“双方只想知道交集，不想暴露交集之外的内容”。例如两个 Agent 或两个机构要确认是否有共同实体、共同风险对象、共同候选文档，PSI 可以只暴露交集。

第四，ABE，属性基加密。它解决的是细粒度访问控制。数据不是简单地加密给某一个人，而是按照属性策略加密，比如“项目成员 + 审计权限 + 当前有效身份”。满足策略才能解密，不满足策略时权限系统即使出错，也无法直接拿到明文。

第五，FHE，全同态加密。它的能力最强，可以在密文上计算函数，最后解密得到正确结果。它适合表达“数据不出密文状态，但仍然可以计算”。代价也最高，所以后面我会把它放在高价值、低频次的场景里。

第六，MPC，安全多方计算。它解决的是多方共同计算的问题。各方不交出原始输入，但可以共同得到函数结果。相比 FHE，它更强调多方协议；相比单方加密，它更适合机构协作、跨平台协作。

<div class="al-kicker al-kicker--proof">缺口① · 可验证性</div>

<div class="al-flow al-flow--sm" style="justify-content:flex-start;">

</div>

<div class="al-cite" v-click="4">

<!--
第一个具体场景是让推理结果可验证，对应零知识证明。

先说问题。现在很多模型能力是以服务形式提供的，用户只能看到输入和输出，看不到服务端到底用了哪个模型，也看不到中间是否按约定执行。比如服务商承诺使用某个大模型完成推理，但用户没有办法判断它是否换成了更小的模型；再比如 Agent 在调用工具之前生成了一组参数，用户也很难确认这些参数是不是来自合规的推理过程。

零知识证明可以把这个问题改写成一个可验证关系。左边这条线表示：模型方有输入 x，也有保密的模型权重 w，经过推理函数 f，得到输出 y，同时生成一个证明 π。验证者拿到的是 x、y、π 和验证公钥 vk。公式的含义是：Verify 输出 1，当且仅当存在某个权重 w，使得 y 等于 f(x,w)。

这里最关键的一点是，w 不需要公开。模型方可以证明“这个结果确实来自某个承诺模型”，但不需要把模型权重交给用户。这样同时解决两个问题：用户获得可验证性，模型方保留知识产权。

放到 Agent 里，最直接的课题不是证明整个 LLM 的完整推理。完整大模型太大，全流程证明目前代价很高。更现实的方向是选择性证明，也就是只证明少数高风险节点。例如，Agent 最终选择了哪个工具，生成了什么工具参数，是否触发了转账、删除、写文件这类敏感动作。只要这些节点能被证明，系统就不需要完全相信模型的自我声明。

选择性证明可以再拆成三层。第一层是证明输入和模型版本没有被替换，例如用承诺固定模型版本、系统提示词和关键上下文。第二层是证明某个规则被执行，例如“只有检索结果中出现过的条款才能被引用”。第三层是证明动作满足安全策略，例如“金额超过阈值时必须经过人工确认”。这样做的好处是，证明对象不再是整个大模型，而是 Agent 链路里少数可以形式化的判断。

所以这一页的研究判断是：ZKP 在 Agent 里的短期价值，不是“证明整个大模型”，而是“证明关键决策节点”。这会把问题规模降下来，也更接近工程可用。

<div class="al-cite">

<!--
第二个具体场景，是让 Agent 的数据接入隐私化。这里对应的原语包括 SSE、PSI 和 ABE，其中我认为最值得展开的是 RAG + SSE。

先说明为什么 RAG 是关键。Agent 如果只依赖模型参数内部的知识，能力会很受限。实际系统里，Agent 往往要接入企业知识库、个人文档、历史会话和工具返回结果，再把检索到的内容交给模型。也就是说，RAG 不是一个边缘能力，而是 Agent 获得外部知识的基础链路。

但 RAG 的隐私问题也很直接。用户的 query 往往不是普通关键词，而是完整意图。例如，“帮我分析这份合同有没有风险”，“结合我的病历解释这个指标”，“根据公司内部文档生成投标方案”。这些 query 本身就可能包含敏感信息。再进一步，知识库里的文档也可能是企业内部资料、个人文件或业务数据。如果检索服务由外部平台提供，query 和文档内容就会进入一个跨信任域接口。

SSE 的作用，就是把“让服务器帮我检索”和“服务器看到明文”这两件事拆开。用户先把 query 转成搜索令牌 token，再用这个 token 查询密文索引 EDB。服务器可以根据 token 和密文数据库返回相关文档 id，但它看到的是密文，而不是明文 query 和明文文档。

这里需要注意，经典 SSE 更多面向关键词检索，而 Agent RAG 更常见的是语义检索、向量近邻和多轮上下文检索。这正是研究空白所在。问题不是简单地把现成 SSE 拿过来，而是要回答：语义相似性怎么在密态下表达，向量近邻怎么和密文索引结合，SSE 的 leakage 在语义检索里应该怎么定义和量化。

如果把这个方向进一步写成课题，我认为至少要回答三个问题。第一，检索质量怎么评估，也就是密态检索之后的召回率、准确率和明文 RAG 相比损失多少。第二，隐私泄露怎么定义，服务器到底能看到访问模式、结果大小，还是还能推断查询相似性。第三，系统开销怎么控制，包括索引构建时间、单次查询延迟、通信量和密钥管理成本。只有同时回答这三类问题，RAG + SSE 才不是简单拼接，而是一个可评估的系统方案。

右边三类原语可以分工。SSE 负责密文检索，保护 query 和文档；PSI 适合跨库匹配，比如两个知识源只想知道共同候选，不想暴露完整集合；ABE 适合细粒度授权，比如不同 Agent、不同工具、不同用户只能解密满足策略的文档。

所以我对这一页的判断是：如果只能先选一个方向，我会优先选输入隐私，尤其是 RAG + SSE。理由有三点。第一，需求真实，Agent 系统确实频繁检索外部数据。第二，工具成熟，隐私检索已有多年积累。第三，空白明确，经典 SSE 和现代语义检索之间还没有完全打通。这个方向既不是纯工程，也不是为了套用密码学而构造问题，而是能把密码学问题放进一个真实 Agent 链路里。

## 选题方法论：四步把想法落成课题

<div class="al-kicker">方法论 · 四步法</div>

<div class="al-steps mt-5">
  <div class="al-step"><div class="idx">1</div><div><div class="t">先找信任缺口</div><div class="d">在 Agent 链路上，定位"系统不得不信任某个对象"的那个点</div></div></div>
  <div class="al-step"><div class="idx">2</div><div><div class="t">再选密码学原语</div><div class="d">按缺口类型，从四组里挑，避免从原语反推问题</div></div></div>
  <div class="al-step"><div class="idx">3</div><div><div class="t">定威胁模型与假设</div><div class="d">敌手能力是什么？困难假设是什么？泄露上界是多少？</div></div></div>
  <div class="al-step"><div class="idx">4</div><div><div class="t">同时评估安全与开销</div><div class="d">不只报安全性，还要报延迟、通信、可用性——这决定能不能落地</div></div></div>
</div>

<div class="al-callout mt-6" v-click="1">
一句话：<b>从缺口出发，不要从原语出发</b>。先想清楚保护什么、防谁，再选工具。否则容易形成"看起来很密码学，但没有解决真实问题"的方案。
</div>

<!--
这一页不是讲某个具体原语，而是把前面的方向判断变成选题方法。因为“AI + 密码学”这个题很大，如果没有方法，很容易变成单纯罗列原语，或者把熟悉的工具套到一个并不真实的问题上。

第一步，先找信任缺口。不要先问“我会 ZKP，能不能用到 Agent 里”，而要先沿着 Agent 链路问：系统在哪个位置不得不信任某个对象。比如不得不信模型服务商确实用了某个模型，不得不信检索服务没有查看 query，不得不信权限系统没有被绕过。这个“不得不信”的位置，才是密码学可能发挥作用的接口。

第二步，再选密码学原语。缺口不同，工具也不同。可验证性问题优先看 ZKP、承诺和签名；输入隐私问题优先看 SSE、FHE；访问控制问题可以看 ABE、PSI；多方互信问题可以看 MPC、FHE。顺序不能反过来。先有缺口，再有原语，而不是先有原语，再去寻找应用。

第三步，明确威胁模型和假设。密码学方案必须说清楚防谁。是防半诚实服务端，还是防恶意服务端？服务端是否知道嵌入模型？是否能保存历史查询？是否有辅助语料？以 SSE 为例，还要说清楚 leakage：访问模式、搜索模式、结果大小、时间模式，哪些泄露允许，哪些泄露不允许。如果这些不讲清楚，所谓安全性就没有可比较的基础。

第四步，同时评估安全和开销。密码学论文容易只强调安全性质，但 Agent 系统是否能落地，还取决于延迟、通信量、存储、吞吐和可用性。一个方案如果每次查询需要几分钟，或者每次工具调用都要巨大证明，就很难进入真实系统。

用 RAG + SSE 举例，四步可以这样落下去。第一，缺口是用户 query 和知识库内容会暴露给检索服务。第二，原语优先选择 SSE，再根据是否需要跨库匹配引入 PSI，根据是否需要细粒度授权引入 ABE。第三，威胁模型可以先从半诚实服务器开始，再逐步考虑恶意服务器和自适应查询攻击。第四，评价时同时报告召回率、查询延迟、索引大小和 leakage。这样，一个方向就从“AI + 密码学”的大标题，变成了可以写问题定义、系统设计和实验评估的课题。

所以这一页可以归纳成一句话：从缺口出发，不从原语出发。先说明保护什么、防谁、允许泄露什么，再选择合适工具，并同时说明代价是否可接受。

## 边界与代价：密码学不是银弹

<div class="al-kicker">务实 · 三条边界</div>

<div class="al-grid3 mt-6">
  <div class="al-card al-card--muted" v-click="1">
    <div class="hd" style="color:var(--al-ink-soft);">开销</div>
    <div class="text-xs mt-1" style="color:var(--al-ink-soft);line-height:1.65;">FHE / ZKP 仍然昂贵，SSE 有可量化的 leakage。能力越强，代价越高。</div>
  </div>
  <div class="al-card al-card--muted" v-click="2">
    <div class="hd" style="color:var(--al-ink-soft);">假设</div>
    <div class="text-xs mt-1" style="color:var(--al-ink-soft);line-height:1.65;">保证依赖威胁模型与困难假设。模型变化或假设不成立，保证随之失效。</div>
  </div>
  <div class="al-card al-card--muted" v-click="3">
    <div class="hd" style="color:var(--al-ink-soft);">不覆盖的问题</div>
    <div class="text-xs mt-1" style="color:var(--al-ink-soft);line-height:1.65;">prompt 注入的语义层、模型对齐、工具的真实副作用，密码学都不负责。</div>
  </div>
</div>

<div class="al-callout mt-7" v-click="4">
合理策略是<b>混合与选择性</b>：只在跨信任域的关键接口上用密码学硬保证，其余交给工程手段。密码学补的是"信任"这一类缺口，不是全部。
</div>

<!--
这一页讲边界。前面一直在说密码学能给 Agent 提供更强的保证，但这里必须明确：密码学不是银弹，它不能替代 Agent 安全的全部工程。

第一条边界是开销。FHE 和 ZKP 目前仍然昂贵，尤其是面对大模型推理这种复杂计算时，完整证明或完整密态计算的成本很高。SSE 相对成熟，也更接近落地，但它通常会有 leakage，比如访问模式、搜索模式、结果大小。这些泄露可以被定义、量化和控制，但很难完全消失。总体规律是：表达能力越强，代价越高；保证越硬，系统复杂度越高。

第二条边界是假设。密码学的保证不是凭空来的，它依赖威胁模型和困难假设。比如半诚实模型和恶意模型完全不同；服务端是否知道嵌入模型，会影响隐私评估；攻击者是否能保存历史查询，也会影响 leakage 的含义。假设一变，结论也要重新评估。

第三条边界是适用范围。prompt 注入属于语义层问题，模型对齐属于目标和行为问题，工具执行的真实副作用属于系统工程问题。这些问题不能直接靠密码学解决。比如密码学可以证明某个工具参数确实来自某个流程，但它不能保证这个参数在现实世界里一定不会造成业务损失。这个判断仍然需要工程规则和业务逻辑。

所以合理策略是混合和选择性使用。在跨信任域的关键接口上，用密码学提供硬保证；在系统内部控制、模型行为约束、工具执行安全这些地方，继续依赖工程手段。密码学补的是“信任”这一类缺口，不是 Agent 系统的全部问题。

这也意味着，做方案时不能只给密码学协议，还要给工程基线。比如没有 SSE 时，普通 RAG 的风险和性能是什么；加入 SSE 以后，隐私多了哪些保证，性能损失是多少；哪些风险仍然要靠权限系统、日志和人工确认解决。只有把这些边界说清楚，方案才不会显得夸大。

这也是为什么我前面一直强调选接口，而不是把所有 Agent 安全问题都密码学化。

<!--
最后收束成一个研究判断。

第一，主线是：Agent 把 AI 工程化、接口化。大模型本身很难直接被密码学处理，因为它太大、太复杂、太不确定。但 Agent 把模型放进了一个有工具、有数据、有权限、有循环的系统里。系统一旦有边界，密码学就有可能在边界上发挥作用。

第二，框架是：四阶段演进，暴露四类信任缺口，对应四组密码学原语。语言模型带来可验证性问题，Agent 带来行动可信问题，Harness 带来数据隐私和访问控制问题，Loop Engineering 带来多方互信问题。每一类缺口都能对应一组原语，也都能进一步展开成具体课题。

第三，我的判断是：未来值得做的，不是把整个大模型强行放进密码学协议，而是在 Agent 的信任边界上使用密码学。谁能先把某一类缺口对应的原语做到工程可用，谁就能在这个方向上形成优势。

第四，如果从我自己的方向出发，我会优先选择 RAG + 可搜索加密。原因前面讲过：需求真实，Agent 检索是高频链路；工具已有基础，SSE 和隐私检索不是从零开始；空白明确，语义检索和密态索引之间还没有完全打通。相比完整 LLM 证明或大规模 FHE 推理，这个方向更接近可落地研究。

这也解释了首页的选题动机。今天虽然讲的是 Agent，但它不是偏离原方向，而是把 RAG 和 SSE 放到一个更大的系统背景里看。Agent 让检索、工具、权限和多轮记忆连成了真实链路，也让原本相对独立的隐私检索问题变得更有应用场景。因此，这个题的重点不是追 AI 热点，而是判断密码学在新系统形态下应该进入哪个接口。
